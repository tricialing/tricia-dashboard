Web Dev · Learning
Mar 2026 – ongoing 5 phases · self-paced Vehicle: this dashboard

The dashboard is the curriculum. No tutorials, no toy projects — every concept here was lived through building something real with real constraints. You already have instincts. This gives them names.

The goal is not to become a developer. It is to understand enough to have strong opinions, make good decisions, and know when to ask the right question. That's a designer with web literacy — which is what you already are, just more so.

Phase 1 · HTML & CSS Fundamentals
The building blocks. You already read this code and have opinions on it. The gap is vocabulary and confidence.
1.1
HTML is structure, not style
Every element is a box. div, p, span, header, footer — they describe what something is, not how it looks. You have been reading and editing these correctly all along. Lived: every page in this dashboard

1.2
The box model — margin, padding, border
Everything on screen is a rectangle. Padding is space inside the box. Margin is space outside. Border is the edge. You already think in these terms from design school. Lived: .box padding in every page

1.3
CSS selectors and specificity
How CSS decides which rule wins when two rules fight over the same element. Class beats tag. ID beats class. Inline beats everything. This is why order and structure matter. Lived: shared CSS loading before page styles
concept
✓
1.4
Flexbox — one-dimensional layout
Flex lets you line things up in a row or column and control how they space and wrap. The nav, header-meta, progress panel, footer — all flexbox. Lived: .nav, footer, .progress-panel
concept
✓
1.5
CSS Grid — two-dimensional layout
Grid lets you define rows and columns at the same time. More powerful than flexbox for page layouts. The week rows, the tools grid on the dashboard, the arc track — all grid. Lived: .week-row, .tools-grid, .top-row

CSS variables (custom properties)
Define a value once, use it everywhere. --ink, --muted, --rule. Change one line, the whole site updates. This is why we have listening-season.css — it's the single source of truth for colour and spacing. Lived: :root tokens in listening-season.css

Phase 2 · Separation of Concerns · DRY
You lived through this refactor this weekend. Now give it the right names.
2.1
HTML / CSS / JS — three jobs, three files
Structure, style, behaviour. Mixing them together works until it doesn't. Separating them means changing one thing doesn't break another. This is the single most important concept in front-end architecture. Lived: extracting listening-season.css and .js
principle
○
2.2
DRY — Don't Repeat Yourself
If you write the same thing twice, you will eventually change one and forget the other. The shared CSS is DRY in practice. Every page was re-declaring body and font — that was the problem we fixed this session. Lived: the entire cleanup pass Mar 10 2026
principle
○
2.3
Cascade order — why the order of your CSS links matters
CSS reads top to bottom. A rule lower in the file wins over the same rule higher up. This is why shared CSS must load before page-specific styles — so page rules can override the shared ones intentionally. Lived: fixing wrong-order links in 5 files
concept
○
2.4
Dev vs production
Dev is where you build and experiment. Production is what real users see (your Netlify site). The base template workflow — build self-contained, do a production pass before pushing — is a lightweight version of this distinction. Lived: base-template.html production pass workflow
concept

Phase 3 · JavaScript Basics
JS is behaviour — it makes things happen. You have already seen it work. Now understand what it's actually doing.
3.1
What JS actually does — the DOM
JS talks to the page through the DOM (Document Object Model) — a map of every element on the page. JS can find elements, read them, change them, add or remove them. Every toggle and countdown is JS doing this. Lived: checkbox toggles in anatomy-prep.html
concept
○
3.2
Variables, functions, events
Variables hold values. Functions are reusable instructions. Events are triggers — a click, a page load, a timer firing. Almost every JS pattern is a combination of these three things. Lived: toggleDone(), onclick, the TERMS array in listening-season.js
concept
○
3.3
localStorage — making things persist
localStorage lets a page remember things after you close it. That's why your anatomy checkboxes stay ticked between visits — they're saved to your browser, not a server. It's local to your device. Lived: checkbox state in anatomy-prep.html
concept
○
3.4
Dates and countdowns
JS has a built-in Date object that knows what time it is right now. Subtract two dates, convert to days or weeks. The school countdown, the solar term logic — all date math. Lived: countdowns in index.html, solar term logic in listening-season.js
concept

Phase 4 · Git & Deployment
You are already doing this. Understanding the mental model makes it less fragile.
4.1
What Git actually is
Git is a time machine for your files. Every commit is a snapshot you can go back to. The repo is the full history of every snapshot. You are not just saving files — you are saving a record of every change and why. Lived: every push to tricialing/tricia-dashboard
concept
○
4.2
Commit messages — why they matter
A good commit message is a note to your future self. Present tense, specific, short. "CSS cleanup pass, fix ordering" tells you exactly what changed and why. "update files" tells you nothing. Lived: commit messages at every session wrap up
practice
○
4.3
How Netlify deployment works
Netlify watches your GitHub repo. Every time you push to main, it pulls the latest files and serves them at your URL. That's the whole pipeline. No servers to manage, no build step yet — just files going live automatically. Lived: tricia-dashboard.netlify.app auto-deploying
concept
○
Phase 5 · Beyond Static HTML
When the dashboard outgrows flat files. Not urgent — but worth knowing what's ahead.
5.1
What a build step is
Right now your files go straight from editor to browser. A build step is a process that transforms your files first — combining them, compressing them, checking for errors. It adds capability but also complexity. You don't need it yet.
concept
○
5.2
Components — stop repeating your HTML
Right now the nav and footer are copy-pasted into every file. A component system lets you define them once and reuse them. This is the next logical evolution of DRY for HTML. Frameworks like React or even simple tools like web components solve this.
concept
○
5.3
When to reach for a framework
Frameworks solve specific scaling problems — repeated UI, shared state, complex interactions. They also add learning overhead and dependency. The honest question is always: does the problem I have right now actually require this? For the dashboard today, probably not yet.
concept
○
Phase 6 · Backend & Sync · Eventually
When localStorage isn't enough. This is how data lives on a server and follows you across devices.
6.1
What a backend actually is
Right now your site has no memory on a server — it's just files. A backend is a program running on a server that can receive data, store it in a database, and send it back. It's the difference between a brochure and an app.
concept
○
6.2
What a database is
A database is organised storage on a server. Instead of saving checkbox state to your browser (localStorage), you save it to a database. Now any device can read it. This is how syncing works everywhere — Capacities, Notion, all of it.
concept
○
6.3
APIs — how front and back talk to each other
An API is a contract between two programs. Your front-end sends a request ("save this checkbox state"), the backend receives it, stores it, and sends back a response ("saved"). Every app you use is doing this constantly in the background.
concept
○
6.4
First backend project — synced checkboxes
The practical application of everything above. Take the anatomy-prep or webdev-learning checkboxes and make them sync across your phone and laptop. Small project, real problem, concrete payoff. This is the right first backend build.
